# Harness engineering: 在 agent-first 世界中利用 Codex

> 原文: https://openai.com/index/harness-engineering/
> 作者: Ryan Lopopolo
> 日期: 2026-02-11

---

过去五个月里，我们团队一直在做一项实验：在完全不手写任何代码的前提下，构建并发布一个软件产品的内部测试版。

这个产品已经有内部日活用户和外部 alpha 测试者。它会发布、部署、出故障，也会被修复。不同之处在于，其中每一行代码, 包括应用逻辑、测试、CI 配置、文档、可观测性和内部工具，全部都由 Codex 编写。我们估计，构建它所花的时间大约只有手写代码的 1/10。

人类负责掌舵，agent 负责执行。

我们之所以刻意加上这个约束，是因为我们想真正构建出那些能够把工程速度提升几个数量级所必需的东西。我们只有几周时间，却要交付最终达到百万行代码规模的系统。为做到这一点，我们必须理解：当一个软件工程团队的主要工作不再是写代码，而是设计环境、明确意图、建立反馈回路，让 Codex agent 能稳定完成工作时，什么会发生变化。

这篇文章讲的是，我们在用一支 agent 团队从零打造一款全新产品时学到了什么，哪些地方出过问题，哪些能力产生了复利，以及如何最大化利用我们唯一真正稀缺的资源：人类的时间与注意力。

## 我们从一个空的 git 仓库开始

第一次向这个空仓库提交代码是在 2025 年 8 月下旬。

最初的脚手架，包括仓库结构、CI 配置、格式化规则、包管理器设置以及应用框架，都是由 Codex CLI 基于 GPT-5 生成的，并参考了一小组现有模板。甚至连指导 agent 如何在仓库中工作的初始 `AGENTS.md` 文件，都是由 Codex 自己写的。

系统一开始并没有任何现成的人类手写代码可供依托。从第一天起，这个仓库就是围绕 agent 来塑造的。

五个月后，这个仓库已经包含了约百万行代码，覆盖应用逻辑、基础设施、工具、文档和内部开发者实用程序。在这段时间里，仅由三位工程师驱动 Codex，就发起并合并了大约 1,500 个 pull request。换算下来，平均每位工程师每天产出 3.5 个 PR。令人意外的是，随着团队增长到如今的七位工程师，吞吐量反而还在提升。

更重要的是，这并不是为了产出而产出：这个产品已经被数百名内部用户使用，其中包括每天都深度使用它的内部重度用户。

在整个开发过程中，人类从未直接贡献任何代码。这也逐渐成为团队的核心理念：不手写代码。

## 重新定义工程师的角色

缺少人类亲自编码之后，工程工作变成了另一种形式，重点转向系统、脚手架和杠杆效应。

早期进展比我们预想得更慢，不是因为 Codex 做不到，而是因为环境定义得不够充分。agent 缺少完成高层目标所需的工具、抽象和内部结构。于是，我们工程团队的主要职责变成了让 agent 具备做出有用工作的能力。

在实践中，这意味着采用深度优先的方式推进：把更大的目标拆解成更小的构件(设计、代码、评审、测试等)，提示 agent 去构建这些构件，再用它们去解锁更复杂的任务。

当某件事失败时，修复方式几乎从来不是“再努力一点”。因为唯一能推动进展的方法，就是让 Codex 把工作做完，所以人类工程师总是会进入这个任务并追问：“缺了什么能力？我们如何把它变成既便于 agent 理解，又能被强制执行的东西？”

人类几乎完全通过 prompt 与系统交互：工程师描述一个任务，运行 agent，让它发起 pull request。为了把一个 PR 推进到完成，我们会指示 Codex 在本地审查自己的改动，在本地和云端请求额外的、针对性的 agent 评审，响应任何来自人类或 agent 的反馈，并在一个循环里持续迭代，直到所有 agent reviewer 都满意为止(本质上这就是一个 Ralph Wiggum Loop)。

Codex 会直接使用我们的标准开发工具(如 `gh`、本地脚本，以及内嵌在仓库中的 skill)，自行收集上下文，而不需要人类把内容复制粘贴进 CLI。

人类可以审查 pull request，但并非必须。随着时间推移，我们已经把几乎所有评审工作都推向了 agent 与 agent 之间完成。

## 提高应用对 agent 的可读性

随着代码吞吐量上升，我们的瓶颈变成了人类 QA 能力。由于固定约束始终是人类的时间与注意力，我们一直在努力给 agent 增加更多能力，让应用 UI、日志和应用指标这些对象本身，都能被 Codex 直接读取和理解。

例如，我们让应用能够按 git worktree 启动，这样 Codex 就能为每个改动启动并驱动一个独立实例。我们还把 Chrome DevTools Protocol 接入了 agent 运行时，并创建了用于处理 DOM 快照、截图和页面导航的 skill。这样一来，Codex 就能直接复现 bug、验证修复，并推理 UI 行为。

![图示：Codex 通过 Chrome DevTools MCP 驱动应用，以验证其工作结果。](https://images.ctfassets.net/kftzwdyauwt9/1Gu58eNlqDEuITmbqJDmq9/1e2e62f7e15fb16d2da0da5407240564/fig_1__codex_drives_the_app_.png?fm=webp&q=90&w=3840)

我们对可观测性工具也做了同样的事情。日志、指标和 trace 通过本地可观测性栈暴露给 Codex，而且这个栈会针对每个 worktree 临时创建。Codex 操作的是该应用的一个完全隔离版本，包括它自己的日志和指标，而这些内容会在任务完成后被销毁。agent 可以用 LogQL 查询日志，用 PromQL 查询指标。

有了这些上下文之后，像“确保服务启动在 800ms 内完成”或者“这四条关键用户路径中的任何 span 都不能超过两秒”这样的 prompt，就变得可操作了。

![图示：在本地开发环境中为 Codex 提供完整的可观测性栈。](https://images.ctfassets.net/kftzwdyauwt9/7uWHsJIC6o3uQPsnQ2Avz9/8be3e321892054bd215afb2b250a176a/OAI_Harness_engineering_The_limits_of_agent_knowledge_desktop-light.png?fm=webp&q=90&w=3840)

我们经常会看到，单次 Codex 运行会在同一个任务上持续工作超过六小时(很多时候还是在人类睡觉时进行)。

## 我们把仓库知识变成了事实来源系统

上下文管理，是让 agent 在大型复杂任务中高效工作的最大挑战之一。我们学到的最早一条经验其实很简单：给 Codex 一张地图，而不是一本 1,000 页的说明书。

我们试过“一个巨大的 `AGENTS.md`”这种做法。结果它以可预见的方式失败了：

* 上下文是稀缺资源。一个巨大的说明文件会挤占任务、代码和相关文档的上下文空间，导致 agent 要么漏掉关键约束，要么开始围绕错误目标优化。
* 过多的指导就等于没有指导。当所有东西都“很重要”时，就没有任何东西真正重要。agent 最终只会在局部做模式匹配，而不是有意识地导航。
* 它会很快腐化。一个单体手册最终会变成一片过时规则的墓地。agent 无法判断哪些内容仍然有效，人类也不再维护它，这个文件会悄悄变成一个极具误导性的负担。
* 它很难验证。单一大块内容不利于机械化检查(覆盖率、时效性、负责人、交叉链接)，因此漂移不可避免。

所以，我们不再把 `AGENTS.md` 当成百科全书，而是把它当成目录。

仓库的知识库位于一个结构化的 `docs/` 目录中，并被视为事实来源系统。一个简短的 `AGENTS.md`(大约 100 行) 会被注入上下文，主要承担地图的作用，把 agent 指向更深层的真实信息来源。

#### 纯文本

```text
AGENTS.md
ARCHITECTURE.md
docs/
├── design-docs/
│   ├── index.md
│   ├── core-beliefs.md
│   └── ...
├── exec-plans/
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/
│   └── db-schema.md
├── product-specs/
│   ├── index.md
│   ├── new-user-onboarding.md
│   └── ...
├── references/
│   ├── design-system-reference-llms.txt
│   ├── nixpacks-llms.txt
│   ├── uv-llms.txt
│   └── ...
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

仓库内知识存储的布局如上。设计文档会被编目和索引，其中包括验证状态，以及一组定义 agent-first 运作原则的核心信念。Architecture documentation 提供了领域和包分层的顶层地图。质量文档会为每个产品域和架构层打分，并持续跟踪差距。

计划被视为一等工件。轻量级的临时计划用于小型改动，而复杂工作则记录在 execution plans 中，其中包含进度和决策日志，并直接提交进仓库。活跃计划、已完成计划和已知技术债都被版本化并放在一起，使 agent 无需依赖外部上下文也能工作。

这实现了一种渐进式披露(progressive disclosure)：agent 从一个小而稳定的入口开始，然后学会去哪里查找下一步信息，而不是一开始就被上下文淹没。

我们通过机械化方式强制执行这一点。专门的 linter 和 CI 任务会验证知识库是否保持最新、是否有交叉链接、以及结构是否正确。一个周期性运行的“doc-gardening” agent 会扫描那些过时或不再反映真实代码行为的文档，并发起修复用的 pull request。

## 目标是让 agent 能读懂

随着代码库演进，Codex 用来做设计决策的框架也必须一起演进。

由于整个仓库都由 agent 生成，因此它首先针对 Codex 的可读性进行了优化。就像团队会努力让新入职工程师更容易导航代码一样，我们的人类工程师的目标，是让 agent 能够直接通过仓库本身推理完整的业务领域。

从 agent 的视角看，任何在运行时无法进入上下文、无法被访问的信息，实际上就等于不存在。存在于 Google Docs、聊天线程或人脑中的知识，对系统来说都是不可访问的。它能看到的，只有仓库本地、可版本化的工件(例如代码、Markdown、schema、可执行计划)。

![图示：agent 知识的边界。Codex 看不见的东西，就等于不存在。](https://images.ctfassets.net/kftzwdyauwt9/7uWHsJIC6o3uQPsnQ2Avz9/8be3e321892054bd215afb2b250a176a/OAI_Harness_engineering_The_limits_of_agent_knowledge_desktop-light.png?fm=webp&q=90&w=3840)

我们逐渐意识到，必须把越来越多的上下文推入仓库中。那场让团队在某个架构模式上达成一致的 Slack 讨论，如果 agent 无法发现它，那么它对 agent 来说就是不可读的，就像对三个月后新加入的员工来说也是未知的一样。

给 Codex 更多上下文，意味着要组织并暴露正确的信息，让 agent 能对其进行推理，而不是用临时性的指令把它淹没。就像你会向一位新同事介绍产品原则、工程规范和团队文化(连 emoji 偏好也包括在内)一样，把这些信息交给 agent，会让它的输出更加符合预期。

这种视角让很多权衡变得清晰。我们偏好那些可以在仓库内被完整吸收和推理的依赖与抽象。那些常被称作“boring”的技术，通常更容易被 agent 建模，因为它们具有更好的可组合性、API 稳定性，以及在训练数据中的代表性。在某些情况下，比起绕过公共库中那些不透明的上游行为，让 agent 重新实现一小部分功能反而更便宜。

例如，我们没有引入一个通用的 `p-limit` 风格包，而是自己实现了一个带并发控制的 map helper：它与我们的 OpenTelemetry instrumentation 深度集成，拥有 100% 的测试覆盖，并且行为完全符合我们的运行时预期。

把越来越多的系统转化为 agent 可以直接检查、验证和修改的形式，不仅能提升 Codex 的杠杆效应，也能提升其他在同一代码库上工作的 agent(例如 Aardvark)的杠杆效应。

## 强制执行架构与品味

仅靠文档，并不能让一个完全由 agent 生成的代码库保持一致性。通过强制执行不变式，而不是微观管理实现细节，我们让 agent 能高速交付，同时不至于破坏基础。例如，我们要求 Codex 在边界处解析数据形状(parse data shapes at the boundary)，但并不规定必须用哪种方式实现(模型似乎偏好 Zod，但并不是我们指定的)。

agent 在具有严格边界和可预测结构的环境中最有效，因此我们围绕一种刚性的架构模型构建了应用。每个业务域都被划分为一组固定层级，并严格校验依赖方向，只允许有限的边。这些约束通过自定义 linter(当然也是 Codex 生成的)和结构化测试，以机械方式强制执行。

下图展示了这条规则：在每个业务域内部(例如 App Settings)，代码只能按照固定层级“向前”依赖(Types → Config → Repo → Service → Runtime → UI)。横切关注点(auth、connectors、telemetry、feature flags)只能通过一个显式接口进入：Providers。除此之外的任何依赖都被禁止，并通过机械方式强制执行。

![图示：具有显式横切边界的分层领域架构。](https://images.ctfassets.net/kftzwdyauwt9/4Rlip1H3T9apPlSmWs7Wr8/7708c176bfbe11951e06ad8e2b83bf01/OAI_Harness_engineering_Layered_domain_architecture_with_explicit_cross-cutting_boundries_desktop-light.png?fm=webp&q=90&w=3840)

这类架构通常要等到团队有数百位工程师时才会认真推动。但在 coding agent 场景里，它是一个很早就必须满足的前提：这些约束正是我们在不牺牲代码质量、不过早衰败、不过度架构漂移的情况下保持高速的原因。

在实践中，我们通过自定义 linter、结构化测试，以及一小组“品味不变式(taste invariants)”来执行这些规则。例如，我们会静态强制执行结构化日志、schema 与类型的命名约定、文件大小上限，以及平台特定的可靠性要求。由于这些 linter 是自定义的，我们会把修复提示直接写进报错信息里，以便注入 agent 上下文。

在以人类为中心的工作流里，这些规则可能显得吹毛求疵或束手束脚。但对于 agent 而言，它们会变成乘数效应：一旦编码完成，就能在所有地方同时生效。

与此同时，我们也会明确哪些地方约束重要，哪些地方不重要。这很像领导一个大型工程平台组织：边界由中心统一约束，局部则保留自主性。你会非常重视边界、正确性和可复现性；而在这些边界之内，你允许团队，或者说 agent，在表达解决方案时拥有相当大的自由。

最终产出的代码不一定符合人类的风格偏好，但这没关系。只要输出是正确的、可维护的、并且对未来的 agent 运行来说依然可读，就达标了。

人类的品味会被持续地反馈进系统中。评审评论、重构 pull request 和面向用户的 bug，都会被沉淀为文档更新，或者直接编码进工具里。当文档不足以表达约束时，我们就把规则提升为代码。

## 吞吐量改变了合并哲学

随着 Codex 吞吐量上升，许多传统工程规范开始变得适得其反。

这个仓库采用尽量少的阻塞式合并门槛。pull request 生命周期很短。测试波动(test flakes)通常通过后续运行来修复，而不是无限期阻塞进度。在一个 agent 吞吐量远超人类注意力的系统里，纠错很便宜，等待却很昂贵。

如果是在低吞吐环境中，这么做会显得不负责任。但在这里，这往往是正确的权衡。

## “agent-generated” 实际上意味着什么

当我们说这个代码库是由 Codex agent 生成的，我们的意思是：代码库中的一切，都是如此。

agent 产出的内容包括：

* 产品代码和测试
* CI 配置和发布工具
* 内部开发者工具
* 文档和设计历史
* 评估 harness
* 评审评论和回复
* 管理仓库本身的脚本
* 生产仪表盘定义文件

人类始终仍在闭环之中，只不过工作抽象层级和过去不同了。我们负责安排优先级、把用户反馈翻译为验收标准、并验证结果。当 agent 遇到困难时，我们会把它视为一种信号：识别缺失项，可能是工具、护栏或文档，再把它反馈回仓库，而且始终由 Codex 自己来写出修复内容。

agent 会直接使用我们的标准开发工具。它们会拉取评审反馈、逐条回复、推送更新，并且经常会自己 squash 并 merge 自己的 pull request。

## 不断提高的自主性层级

随着越来越多的开发闭环被直接编码进系统，包括测试、验证、评审、处理反馈与故障恢复，这个仓库最近跨过了一个重要门槛：Codex 已经可以端到端地驱动一个新功能。

给它一个 prompt，agent 现在可以：

* 验证代码库当前状态
* 复现一个已报告的 bug
* 录制一段演示故障的视频
* 实现修复
* 通过驱动应用来验证修复
* 录制第二段视频，展示问题已解决
* 发起 pull request
* 响应 agent 和人类的反馈
* 发现并修复构建失败
* 仅在需要判断时升级给人类
* 合并改动

这种行为高度依赖于该仓库的具体结构与工具链，因此不能假设它无需类似投入就能泛化，至少目前还不能。

## 熵与垃圾回收

完全的 agent 自主性也带来了新的问题。Codex 会复制仓库中已经存在的模式，即便这些模式并不均匀，甚至并不理想。随着时间推移，这不可避免地会导致漂移。

最初，人类是手动处理这件事的。我们的团队过去每周五都会花整整一天(一周的 20%)清理“AI slop”。显然，这种方式无法扩展。

于是，我们开始把所谓的“golden principles”直接编码进仓库，并建立了一个周期性清理流程。这些原则是带有明确偏好的机械规则，用来让代码库在未来的 agent 运行中保持可读和一致。例如：(1) 我们更偏好共享 utility package，而不是各写各的 helper，以便把不变式集中管理；(2) 我们不采用“YOLO-style”方式去探测数据，而是在边界处做校验，或依赖类型化 SDK，这样 agent 就不会在猜测的数据形状之上继续构建。

我们会按固定节奏运行一组后台 Codex 任务，扫描偏离这些原则的情况、更新质量评分，并发起有针对性的重构 pull request。这些 PR 大多数都能在一分钟内完成评审并自动合并。

这就像垃圾回收。技术债就像高息贷款：几乎总是持续以小步快跑的方式偿还更划算，而不是让它复利滚大，最后再以痛苦的方式集中处理。人类的品味只需捕捉一次，此后就能持续地应用在每一行代码上。这样我们也能按天发现并纠正坏模式，而不是让它在代码库里传播数天乃至数周。

## 我们仍在学习什么

到目前为止，这套策略在 OpenAI 内部上线和推广阶段运行得不错。为真实用户构建真实产品，帮助我们把这些投入锚定在现实问题上，并引导我们走向长期可维护性。

我们尚不清楚的是，在一个完全由 agent 生成的系统中，架构一致性在多年尺度上会如何演化。我们仍在学习，人类判断究竟在哪些地方最有杠杆效应，以及如何把这种判断编码进去，让它产生复利。我们也不知道，随着模型能力持续增强，这个系统会如何演化。

但有一点已经很清楚：软件构建仍然需要纪律，只不过这种纪律越来越多地体现在脚手架，而不是代码本身。那些让代码库保持一致的工具、抽象和反馈回路，正变得越来越重要。

我们现在最困难的挑战，集中在如何设计环境、反馈回路和控制系统，从而帮助 agent 实现我们的目标：在大规模条件下构建并维护复杂而可靠的软件。

随着像 Codex 这样的 agent 承担软件生命周期中越来越大的比例，这些问题只会变得更加重要。我们希望分享这些早期经验，能帮助你思考应该把精力投入到哪里，这样你就可以专注于构建真正重要的东西。
