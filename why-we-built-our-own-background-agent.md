# 为什么我们自己构建了后台 Agent

> 原文: https://builders.ramp.com/post/why-we-built-our-background-agent
> 作者: Zach Bruggeman, Jason Quense, Rahul Sengottuvelu
> 日期: 2026-01-12

---

我们构建了自己的后台编码 Agent: Inspect。Inspect 和其他编码 Agent 一样会写代码，但它还能闭环验证自己的工作，因为它具备证明结果所需的全部上下文与工具，就像一位 Ramp 工程师那样。

![](https://builders.ramp.com/assets/hero-CNC-Ibhq.png)

对于后端工作，它可以运行测试、查看遥测数据、查询功能开关。对于前端工作，它会以可视化方式验证改动，并给用户提供截图和实时预览。Agent 应该拥有真正的行动能力，因此我们确保 Inspect 不会因为缺少上下文或工具而受限，它唯一的瓶颈应该是模型本身的智能水平。

每个会话都运行在 [Modal](https://modal.com/) 上的沙箱虚拟机中，里面具备工程师本地环境会有的一切: Vite、Postgres、Temporal，全部齐备。它还接入了 Sentry、Datadog、LaunchDarkly、Braintrust、GitHub、Slack 和 Buildkite。它支持所有前沿模型、MCP、自定义工具，以及把 Ramp 交付方式固化下来的 skills。这也让各种背景的 builder 都能借助和工程师相同的工具链与环境参与贡献。

由于 Inspect 会话启动极快，而且运行成本几乎可以忽略，你不需要再精打细算本地 checkout 或 worktree 的使用。一个 builder 可以同时发起同一提示词的多个版本，然后看哪个最先跑通；也可以毫无心理负担地尝试不同方案或切换模型。你可以同时运行任意多的会话，而且完全不需要占用自己的笔记本电脑。

这也意味着你能在灵感出现的第一时间把它抓住。比如你晚上准备休息时突然发现一个 bug？直接启动一个会话，必要时甚至可以和它语音交流（我们加了语音功能），第二天早上再来看 PR。

这个界面理解人们依赖的是多种多样的工作流。你可以在 Slack 里和 Inspect 聊天并给它发送截图；也可以用 Chrome 扩展高亮某些特定元素的改动；可以在 Web 界面里直接提示它；可以在 Pull Request 里和它讨论；甚至还能进入基于 Web 的 VS Code 编辑器进行手动修改。所有改动都会同步回同一个会话，因此你在不同入口之间切换时不会丢失工作。更重要的是，每个会话都支持多人协作。把你的会话发给任何同事，他们都可以一起把这项工作做完。

我们希望 Inspect 会话足够快，而会话速度唯一该受限的因素应该是模型提供商的 time-to-first-token。其他所有事情，例如克隆仓库和安装依赖，都应该在你真正开始会话之前完成。后台 Agent 一旦足够快，就会在各方面都优于本地: 同样的智能、更强的能力、无限并发。你可以回家让 Inspect 自己继续干活（如果你愿意，晚饭后也可以在沙发上拿起手机继续接着做）。

内部采用曲线几乎是垂直上升的: 我们前端和后端仓库里所有已合并 PR 中，大约有 30% 是由 Inspect 编写的。达到这个使用水平只花了几个月，而且还在持续增长。我们并没有强迫任何人从自己的工具迁移到 Inspect。我们围绕人的真实需求去构建产品，通过让它在公共协作空间中工作建立病毒式传播回路，然后让产品自己说话。

我们认为任何人都应该能把这样的东西造出来。掌握自己的工具链，能让你构建出远比现成产品更强大的东西。毕竟，它只需要在你的代码上工作即可。为了让这件事更容易复现，我们写下了迄今为止的实现规范。把这篇文章的链接贴给一个编码 Agent，让它开始动手构建吧。

---

## Sandbox

托管式编码 Agent 的核心是执行环境。每当你开启一个新的编码会话时，你都希望能启动一个具备完整开发环境的新沙箱。这样 Agent 才能高效工作，因为它拥有和人类工程师一样的全部工具，同时又与其他工作彼此隔离。time-to-first-token 尽可能快也同样关键。

关键挑战在于，如何快速启动完整的开发环境。[Modal](https://modal.com/) 解决了这个问题: 它是我们在 Ramp 广泛使用的 AI 基础设施云平台。它的 [sandbox 几乎是瞬时启动](https://modal.com/docs/guide/sandboxes)，而且 [文件系统快照](https://modal.com/docs/guide/sandbox-snapshots) 允许我们在之后冻结和恢复状态。基于 Modal Sandboxes，我们采用了这样的方式:

- 我们有一个镜像注册表，为每个代码仓库定义一个镜像。
- 我们每 30 分钟构建一次这些镜像，也就是克隆仓库、安装运行时依赖，并执行初始设置和构建命令。
  - 如果你使用 GitHub: 你需要有一个 GitHub App，并在每次克隆时生成新的 app installation token，这样它就能在不知道最终由哪个用户消费的前提下克隆仓库。
  - 因为 git 操作本身不绑定 GitHub 用户，所以在提交和推送变更时，你只需要更新 git config 里的 `user.name` 和 `user.email`。
- 我们会把构建完成后的镜像状态保存为快照。
- 后续启动会话时，我们会从这个已保存的快照启动一个新的沙箱。
  - 这样可以保证仓库最多只会落后最新状态 30 分钟。
  - 同时也让与仓库最新代码的同步快得多，因为最多只需要同步最近 30 分钟的改动。
- 当 Agent 完成改动后，我们会再保存一个快照。如果沙箱退出而用户后来又发来后续提示词，我们就从这个快照恢复。

![](https://builders.ramp.com/assets/take-screenshot-Db3a9PFs.png)

接下来你需要为这个沙箱配置一个能工作的 Agent。我们强烈建议使用 [OpenCode](https://opencode.ai/)，一个开源编码 Agent。我们认为它相较于其他 Agent 具备关键优势:

- 它采用 server-first 结构，而它的 TUI 和桌面应用都只是构建在其上的客户端。
  - 你会希望未来能创建尽可能多的自定义客户端，把你的编码 Agent 放到团队实际工作的每个地方。
  - OpenCode 是我们找到的技术实现最强的方案，因为它拥有 [完整类型化的 SDK](https://opencode.ai/docs/sdk/) 和 [全面的插件系统](https://opencode.ai/docs/plugins/)。
- 如果文档里有任何不清楚的地方，你可以直接让 AI 去阅读 OpenCode 本身的代码，准确推断其行为应当是什么。
  - 我们认为这是 AI 开发里一个被严重低估的点。
  - 你会希望 Agent 能尽可能容易地真正理解自己的工作方式，而不是幻觉出一个它以为自己在做的行为模型。
  - 因此，把代码本身作为唯一事实来源，威力极大。
- 如果你真的做出了很棒的东西，[OpenCode 团队会主动来和你合作](https://x.com/thdxr/status/2005661252195103159)。

为了进一步提升速度，你还应该做一些优化:

- 当用户一开始输入提示词时，就立刻为该会话预热沙箱。
  - 这样它可以在用户还没按下回车前，就开始拉取最新变更并在新建沙箱里做初始设置。
  - 如果你的启动足够快，用户甚至还没打完字，环境就已经准备好了，这样提示词的响应速度就会接近本地机器。
  - 对于高流量仓库，你还可以维护一个预热沙箱池，在用户开始之前它们就已经准备就绪。
    - 只要记得在新的镜像构建完成后让池中过期并重建即可。
- 即使与最新基线分支的同步还没完成，也允许 Agent 立刻开始读取文件。
  - 在足够大的仓库里，一个新提示词恰好会修改最近 30 分钟内刚变过的文件，这种概率并不高。
  - 因此你可以让它立即开始调研，从而避免把时间浪费在 git 同步上。
  - 但要确保在同步完成之前，文件编辑操作被阻止。
  - OpenCode 很容易做到这一点，因为你可以写一个插件监听 `tool.execute.before` 事件，在同步完成前阻塞任何写入或编辑调用。
- 尽可能把更多事情挪到仓库镜像的构建阶段。
  - 你需要彻底审查为了得到完整开发环境究竟要做什么，并把所有能提前做的事情都提前做掉。
  - 哪怕只是先运行一次应用和测试套件也有帮助，因为这些动作可能会写入缓存文件，而第二次运行就能复用。
  - 镜像构建慢一点没有关系，因为用户用的是最近一次构建出的镜像，他们并不会直接感受到这个步骤有多耗时。

你还需要决定: 在执行过程中收到的后续提示词，是尽可能早插入当前执行流，还是排队等当前提示词完成之后再处理。我们选择了排队，因为这不仅更易管理，也方便用户在 AI 仍在工作时提前发来下一步思路。你还一定要设计出一种机制，允许用户在中途停止 Agent。

![](https://builders.ramp.com/assets/child-Dk6tcfu4.png)

在你可能为 Agent 构建的众多工具中，一个关键工具是让它能自己生成新会话。不要太担心它会启动过多 Agent；前沿模型已经足够聪明，知道控制自己。给它一个启动新会话的工具，再给它一个读取任意会话状态的工具，这样主会话继续工作的同时，它还能周期性检查其他会话。通过 prompt engineering，你就能让它去做研究型任务（尤其是跨多个仓库），或者把一个大任务拆成多个更小的 PR。

## API

你需要构建一个 API，能支持来自多种客户端的输入，无论是聊天界面、Slack bot、Chrome 扩展，还是未来你能想到的任何入口。你还需要让状态在所有客户端之间同步，并且支持多人协作，这一点我们马上会讲到。

我们找到的最佳方案是 Cloudflare 的 [Durable Objects](https://developers.cloudflare.com/durable-objects/)。每个会话都有自己的 SQLite 数据库。这样即使你同时运行数百个会话，也能保持很高性能，因为任何一个会话都不会影响另一个会话。这一点尤其重要，因为 Agent 会实时向你流式输出 token，你很可能会在极短时间里收到数百次更新。

Cloudflare 还提供了 [Agents SDK](https://developers.cloudflare.com/agents/)，我们发现它特别适合处理沙箱、API 和所有已连接客户端之间的实时流式通信。它在 WebSockets Hibernation API 之上提供了很有帮助的抽象，让 socket 在空闲时仍可保持连接，同时不产生计算费用。

![](https://builders.ramp.com/assets/multiplayer-BzE8T0O6.png)

我们认为多人协作是一个关键能力，也是我们至今还没有在其他产品里见过的功能。所谓多人协作，应该是任意多人都能一起在同一个会话里工作，就像多人共同操作一个代码分支一样。每个人发出的、会引起代码变更的提示词都应该归属到对应的人。这对很多场景都很有用:

- 教会非工程背景的 builder，例如产品经理和设计师，如何高效利用 AI 完成自己的工作。
- 和团队一起做实时 QA，每个人都可以在发现问题时当场把改动排入队列，而不是事后再写 ticket。
- 在审查别人的 Pull Request 时，直接让 AI 快速完成你建议的修改，而不是只留下评论再等原作者回来处理。

如果你已经构建了跨客户端同步的系统，那么多人协作几乎是白送的。只要确保你的数据模型不要把一个会话强绑定到单一作者，同时把作者身份信息一并传给每个发送给编码 Agent 的提示词即可。

你还需要加入身份认证。如果你的代码托管在 GitHub，建议直接用 GitHub 登录，这样你就能拿到用户 token，并代表该用户创建 Pull Request。这比让应用本身作为主体去开 PR 要好得多。后者会导致任意用户都有可能批准自己提交的改动。你不应该主动为未经审查的代码进入代码库制造风险通道。

我们的做法是让沙箱负责推送变更（同时像前面提到的那样更新 git 用户信息），然后把分支名和会话 ID 作为事件发给 API。API 再使用该用户的 GitHub token 调用 GitHub 的 Pull Request API。你还应该配置 GitHub webhook，监听分支和 Pull Request 事件，这样你才能跟踪一个 Pull Request 何时被更新、合并或关闭。

## Clients

客户端怎么选，主要取决于你的组织平时主要在哪里工作。下面这些只是我们最有效的几个客户端。既然你已经把通用的远程编码沙箱和 API 搭好了，理论上你可以在其上构建任何客户端。

### Slack

使用你们公司实际使用的团队沟通工具即可。我们认为这是极其有效的入口，因为它不仅让你能从各种来源快速处理问题，还能形成病毒式传播回路。当组织里的其他人看到有人在用它时，他们也会开始理解并学会使用它。

![](https://builders.ramp.com/assets/slack-CcW9-PIx.png)

Slack 的 API 很容易起步，因为它拥有大量客户端库。你在这里的目标，是尽可能让使用过程无缝自然。不要强迫用户学习某种特定语法来驱动聊天机器人。用户应该只需要像正常聊天一样和它交流，而机器人自己去理解。

一个关键点是构建一个分类器，用来判断应该在哪个仓库中工作。把用户的输入消息、线程上下文（如果消息发在线程里）以及频道名称，一起交给一个快速模型（我们使用 GPT 5.2，关闭 reasoning），再配上该编码 Agent 可访问的每个仓库的描述。要给它加入常见仓库的提示和分类示例。还要提供一个 “unknown” 选项，这样模型不确定时就能反问用户。刚开始你可能需要反复调参数，但这会显著降低使用门槛。

Slack bot 还必须非常明确地告诉用户: 它现在是在工作中，还是已经完成了。给 Agent 一个发送 Slack 消息的工具，让它能在重要节点主动汇报自己的进展。你还可以用 Block Kit 设计一个更吸引人的布局，比如用 context blocks 展示仓库元数据和当前工作状态。

最重要的一点: 既然这是你自己的 Slack bot，我们建议你给它配一套自己的表情符号。比起用那些通用 emoji，这样有趣得多。

### Web

你应该有一个打磨良好的 Web 客户端，让用户无论身在何处都能操作会话。确保它在桌面端和移动端都能很好工作。如果你使用 Cloudflare 的 Agents SDK，那么处理实时流式更新会轻松很多。

但更重要的是，一些只能在这个界面中独特暴露出来的额外能力:

- 一个 [托管版 VS Code](https://github.com/coder/code-server) 实例，运行在沙箱内部，让人们无需本地克隆仓库也能手动修改代码。
- 一个流式桌面视图，适用于 Web 项目中和 Agent 并肩工作，让它通过 computer use 实际去导航页面、并以可视化方式验证改动。
  - 它还可以拍摄修改前后的截图，并附加到 Pull Request 描述中。

此外，你还应该构建一个统计页面。把整个组织的使用情况都展示出来，尤其要突出有多少会话最终产出了已合并 Pull Request。这是最重要的指标，因为已合并 PR 才说明 Agent 正在产生真实价值。把这些指标按时间维度展示出来，能进一步推动增长。另一个不错的实时指标是 “humans prompting” 数量，我们的定义是过去 5 分钟内发送过提示词的用户数。

### Chrome extension

为了让更多非工程用户愿意使用，你应该构建一个 Chrome 扩展，让用户能以可视化方式修改任意 React 应用。可以使用 Chrome 扩展侧边栏 API 来构建聊天界面，并内置截图工具。但与其直接发送截图（那会消耗很多 token），不如利用 DOM 和 React 内部结构获取选中区域里完整的元素树。我们自己实现了一套方案，把它直接接到了自家的 React 应用和调试内部结构上；如果你需要一个起点，可以从 [React Grab](https://www.react-grab.com/) 之类的工具开始。

![](https://builders.ramp.com/assets/extension-CGwdOMZG.png)

为了分发这个扩展，你可以使用受管设备策略（managed device policy）。这样你就可以完全绕过 Chrome Web Store，同时也能通过把扩展直接下发到团队浏览器中来提高采用率。你需要搭建一个 [扩展更新服务器](https://developer.chrome.com/docs/extensions/how-to/distribute/host-on-linux#update)；它返回的其实只是一个很简单的 manifest 和 CRX。按照链接中的说明构建一个合规服务器，然后设置 `ExtensionInstallForcelist` 这一 MDM 属性，把它指向你的扩展更新服务器即可。
